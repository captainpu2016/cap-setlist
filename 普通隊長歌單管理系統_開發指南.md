# 普通隊長歌單管理系統 — 開發指南

> 獨立新專案，與先前開發過的其他東西完全分開（獨立 repo / Supabase project / Vercel project / 網域）

---

## 1. 專案範圍界定

### 1.1 核心目標
- 前台：訪客瀏覽場次，查看每場演出歌單，可自行決定是否生成串流播放清單
- 後台：管理歌曲資料庫、建立與上架各場次歌單

### 1.2 明確排除的範圍（避免規劃時失焦）
- 不做訪客登入 / 會員系統（前台完全公開瀏覽）
- 不串接 Apple Music API（採用純連結制，見第 6.3 節）
- 不做即時互動功能（留言、按讚等不在此專案範圍）
- 不與龐克擂台共用任何資料庫、程式碼、環境變數

---

## 2. 資訊架構（Sitemap）

### 2.1 前台（公開）
```
/                          場次列表（依日期排序，近期在前）
/show/[slug]               單一場次頁：歌單 + 生成播放清單按鈕
```

### 2.2 後台（需登入）
```
/admin/login               管理者登入
/admin                     後台首頁（近期場次總覽 / 待補連結提醒）
/admin/songs               歌曲資料庫列表（搜尋、篩選、新增）
/admin/songs/[id]          單曲編輯（含串流連結欄位）
/admin/shows               場次列表管理
/admin/shows/[id]/setlist  歌單編輯器（選歌、排序、上架開關）
/admin/settings/spotify    Spotify 官方帳號授權設定（一次性設定頁）
```

## 3. 資料庫設計（Supabase / Postgres）

### 3.1 資料表

**songs（歌曲主檔）**
| 欄位 | 型別 | 說明 |
|---|---|---|
| id | uuid, PK | |
| title | text | 歌名 |
| artist | text | 預設「普通隊長」，保留欄位以防未來合作曲目 |
| duration_seconds | int, nullable | 用於歌單總長度計算 |
| spotify_track_id | text, nullable | 純 track ID，不存完整網址 |
| apple_music_url | text, nullable | 完整網址（連結制，非 API ID） |
| youtube_url | text, nullable | 選填 |
| created_at | timestamptz | |
| updated_at | timestamptz | |

**shows（場次）**
| 欄位 | 型別 | 說明 |
|---|---|---|
| id | uuid, PK | |
| slug | text, unique | 前台網址用 |
| title | text | 例：普通隊長十週年專場 |
| show_date | date | |
| venue | text | |
| status | enum: draft / published | 後台編輯完成才「上架」，草稿不會出現在前台 |
| spotify_playlist_url | text, nullable | 快取欄位，見 3.3 節 |
| created_at / updated_at | timestamptz | |

**setlist_items（歌單關聯表）**
| 欄位 | 型別 | 說明 |
|---|---|---|
| id | uuid, PK | |
| show_id | uuid, FK → shows.id | |
| song_id | uuid, FK → songs.id | |
| position | int | 演出順序，用於排序與拖曳 |
| notes | text, nullable | 例：「安可曲」「與 XX 合作」 |
| is_placeholder | boolean, default false | 尚未選定曲目時使用（見 5.2 節） |

**app_settings（系統設定，僅後端讀取）**
| 欄位 | 型別 | 說明 |
|---|---|---|
| key | text, PK | 例：`spotify_refresh_token` |
| value | text（建議加密儲存） | |
| updated_at | timestamptz | |

### 3.2 索引與關聯設計重點
- `setlist_items(show_id, position)` 建複合索引，加速歌單排序查詢
- `songs.spotify_track_id`、`songs.apple_music_url` 允許 null，前端需明確處理「未連結」狀態，不可假設一定有值
- 使用 `slug` 而非 `id` 做前台路由查詢（與龐克擂台既有慣例一致，但這是新專案的獨立慣例，非共用程式碼）

### 3.3 為什麼 shows 表需要 `spotify_playlist_url` 快取欄位
避免訪客每次進頁面都重新呼叫 Spotify API 建立新播放清單。第一次生成後把播放清單網址存回這個欄位，之後訪客進來直接顯示「已有播放清單」+ 開啟連結，右側提供「重新生成」按鈕（歌單異動後才需要）。

### 3.4 Row Level Security 規劃
- `songs`、`shows`、`setlist_items`：公開讀取（`select` for anon），寫入僅限已登入管理者
- `app_settings`：完全不開放給 anon role，只有 service role key（後端）可讀寫

---

## 4. 權限與登入設計

- 後台採用 Supabase Auth，email + password 即可，不需要社群登入
- 建議只建立 1-2 組帳號（你本人 + 協作的工作人員），不做開放註冊
- 所有 `/admin/*` 路由在 middleware 層檢查 session，未登入導回 `/admin/login`
- API Route（尤其是生成播放清單、修改資料庫）都要在伺服器端二次驗證身份，不能只靠前端路由保護

## 5. 後台功能規格

### 5.1 歌曲資料庫管理（/admin/songs）
- 列表頁：搜尋（歌名）、篩選（缺 Spotify / 缺 Apple Music / 全部）
- 新增／編輯表單欄位：歌名、時長、Spotify 連結（貼網址自動解析出 track ID）、Apple Music 連結（直接存網址）、YouTube 連結（選填）
- 貼上網址自動解析邏輯：前端用正則從 `open.spotify.com/track/{id}` 抓出 `{id}`，存純 ID；Apple Music 不解析，直接存完整網址
- 顯示「使用次數」（該曲目被多少場次引用），輔助判斷是否為固定班底曲目

### 5.2 歌單編輯器（/admin/shows/[id]/setlist）
- 左側：從歌曲庫搜尋選曲（輸入框 + 下拉建議，優先顯示常用曲目）
- 右側：目前歌單列表，可拖曳排序、刪除、加註 notes
- 支援「未定曲目佔位」：安可曲尚未決定時，先建一個 `is_placeholder = true` 的項目卡位，前台會標示「敬請期待」而非顯示空白
- 儲存邏輯：歌單項目在主要場次資料存檔後才寫入（避免 show_id 尚未產生時的關聯錯誤）
- 場次狀態切換：`draft` → `published`，只有 published 才會出現在前台 `/`

### 5.3 Spotify 官方帳號授權設定（/admin/settings/spotify，一次性設定）
- 這頁只需要在專案初期設定一次
- 顯示「連接 Spotify 官方帳號」按鈕 → 導向 Spotify OAuth 授權頁（scope: `playlist-modify-public`）
- 授權完成後，後端把拿到的 refresh token 寫入 `app_settings` 表
- 頁面顯示目前連接狀態（已連接 / 未連接 / token 已失效需重新授權）

## 6. 前台功能規格

### 6.1 場次列表（/）
- 依 `show_date` 排序，近期活動在前，過去場次可摺疊或另開「歷史場次」區塊
- 每張卡片顯示：場次名稱、日期、場地、是否已有播放清單（小圖示提示）

### 6.2 單場次頁（/show/[slug]）
- 顯示完整歌單（依 `position` 排序），每首歌顯示歌名、時長
- 若 `is_placeholder`，顯示「敬請期待」而非空白或錯誤
- **Spotify 播放清單生成按鈕**：
  - 若 `shows.spotify_playlist_url` 已存在 → 顯示「開啟 Spotify 播放清單」直接連結
  - 若尚未生成 → 顯示「生成播放清單」按鈕，訪客點擊後由後端呼叫 Spotify API 建立（見 6.4 節流程）
- **Apple Music 區塊**：每首歌旁邊顯示「在 Apple Music 開啟」小連結圖示（純網址跳轉，無需登入授權），沒有連結的歌曲則不顯示圖示

### 6.3 Apple Music 連結制設計理由（重申定案）
不串接 MusicKit API，避免開發者帳號成本與授權流程複雜度。訪客點擊「在 Apple Music 開啟」單曲連結後，自行手動加入播放列表。此設計已與你確認為最終方案。

### 6.4 訪客觸發生成播放清單的流程
1. 訪客在 `/show/[slug]` 按下「生成播放清單」
2. 前端呼叫 `POST /api/shows/[id]/generate-playlist`
3. 後端從 `app_settings` 讀取 Spotify refresh token，換取 access token
4. 呼叫 Spotify API 建立播放清單（用官方帳號名義），依序把歌單中有 `spotify_track_id` 的曲目加入
5. 若歌單中有曲目缺少 `spotify_track_id`，該曲目跳過，回傳結果中附註「有 N 首歌未加入（尚無串流連結）」
6. 成功後把播放清單網址寫回 `shows.spotify_playlist_url`，前端顯示連結並提示訪客可另存到自己帳號

> 注意：因為是官方帳號建立單一份播放清單（非每位訪客各自一份），多位訪客同時按下按鈕時，第二次呼叫應直接回傳已存在的 `spotify_playlist_url`，而非重複建立。後端需做這層判斷。

## 7. 技術棧總結

| 層級 | 選擇 | 備註 |
|---|---|---|
| 前端框架 | Next.js（App Router） | 前後台共用同一專案，用路由區隔 |
| 後端邏輯 | Next.js API Routes / Server Actions | 處理 Spotify OAuth、播放清單生成 |
| 資料庫 | Supabase（全新獨立 project） | Postgres + Auth + RLS |
| 部署 | Vercel（全新獨立 project） | 獨立網域，獨立環境變數 |
| 認證 | Supabase Auth（後台專用） | 前台不需要登入 |
| 外部 API | Spotify Web API | Apple Music 不串 API，純連結 |

## 8. 開發階段規劃（建議順序）

**Phase 1：地基**
- 建立 Supabase project、跑 schema migration（songs / shows / setlist_items / app_settings）
- 設定 RLS policies
- 建立 Next.js 專案骨架、Supabase client 設定、環境變數

**Phase 2：後台基礎 CRUD**
- Supabase Auth 登入頁 + middleware 保護 `/admin/*`
- 歌曲資料庫列表 + 新增/編輯表單
- 場次列表管理 + 新增/編輯表單（含 draft/published 切換）

**Phase 3：歌單編輯器**
- 歌單編輯器 UI（選歌、拖曳排序、notes、placeholder）
- 儲存邏輯與 position 排序處理

**Phase 4：前台頁面**
- 場次列表頁
- 單場次歌單頁（含 Apple Music 逐首連結顯示）

**Phase 5：Spotify 整合**
- Spotify OAuth 授權設定頁（後台一次性設定）
- `/api/shows/[id]/generate-playlist` API Route
- 前台生成播放清單按鈕串接、已存在播放清單的快取顯示邏輯

**Phase 6：收尾**
- 空狀態、錯誤狀態設計（例如歌單全空、Spotify token 失效等情境）
- 基本 SEO（meta tags、Open Graph，方便分享單場次頁到社群）
- 上線前資料檢查（是否所有已上架場次都有正確標示 published）
